Harness com Memory, Skills, MCP, Tools & Loop sobre OpenRouter

Plano técnico de implementação para um harness multi-tenant com gestão de janela de contexto baseada em orçamento de tokens, sumarização encadeada, checkpoints de assunto, tools indexadas via MCP, skills por usuário com indução automática e observabilidade de tokens por turn. Foco na lógica do harness; o Bash executa atrás de uma interface SandboxBackend plugável.

00Visão geral

Sete blocos, um banco. Cada turn é uma mensagem; cada mensagem registra tokens; cada decisão fica gravada no Postgres.

🔁Loop

Ciclo agêntico com controle de orçamento de tokens por session.

🧠Memory

Janela = contexto − system prompt; summary encadeado ao atingir o teto.

🛠️Tools

Built-in + Index Tools (MCP) descobertas por busca, não no prompt.

📦Skills

Por usuário no DB; indução automática a cada 10 sessions.

🔌MCP

Servidores MCP alimentam o índice de tools, não o system prompt.

🌐Provider

OpenRouter: metadata de modelo + chat completions + tool calling.

📊Observabilidade

Todo step logado; tokens_in/tokens_out por turn.

🗄️Postgres

Substrato único: turns, sessions, summaries, checkpoints, skills, tool index.

01Decisões travadas

Premissas confirmadas que guiam todo o plano.

TurnUma mensagem com role (user/system/assistant/tool). Cada turn grava tokens de entrada e saída.
SessionJanela explícita de início e fim. A cada 10 sessions do usuário, roda a indução de skills.
CheckpointA cada 20 turns de role user, classifica o assunto anterior em poucas palavras.
MemoryOrçamento = janela do modelo − system prompt. Memory = mensagens + summaries.
SummaryAo atingir o teto: copia as 10% últimas mensagens, gera summary encadeado de todo o resto. Janela = system + summary + 10%.
Limite tokensTeto por session. Ao atingir, o loop para e retorna a resposta parcial.
ToolsBuilt-in (GetTools, SearchTools, GetSkills, Bash) + Index Tools via MCP, buscadas sob demanda.
ProviderOpenRouter. Stack Python. Persistência em Postgres. Bash atrás de SandboxBackend plugável.

03Modelo de dados (Postgres)

Esquema mínimo que sustenta memory, checkpoints, skills, tool index e contabilidade de tokens. Multi-tenant por user_id em todas as tabelas.

CREATE TABLE users (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  external_id   text UNIQUE NOT NULL,
  created_at    timestamptz DEFAULT now()
);

CREATE TABLE sessions (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       uuid REFERENCES users(id),
  model         text NOT NULL,              -- slug OpenRouter, ex. "anthropic/claude-..."
  context_window int NOT NULL,             -- vindo do metadata do modelo
  token_budget  int NOT NULL,               -- teto de gasto por session
  tokens_spent  int DEFAULT 0,               -- acumulado in+out
  status        text DEFAULT 'open',         -- open | closed | budget_exhausted
  started_at    timestamptz DEFAULT now(),
  ended_at      timestamptz
);

-- cada TURN é uma mensagem; é a unidade central da memory
CREATE TABLE turns (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    uuid REFERENCES sessions(id),
  user_id       uuid REFERENCES users(id),
  idx           int NOT NULL,               -- ordem dentro da session
  role          text NOT NULL,              -- user | system | assistant | tool
  content       jsonb NOT NULL,             -- texto e/ou tool_calls / tool_result
  token_count   int NOT NULL,               -- custo do conteúdo desta mensagem
  tokens_in     int,                          -- prompt tokens (em turns gerados pelo modelo)
  tokens_out    int,                          -- completion tokens
  in_window     bool DEFAULT true,            -- ainda na janela ativa? (false após sumarizar)
  created_at    timestamptz DEFAULT now()
);

-- summaries encadeados: cada um engloba o anterior + as mensagens evictadas
CREATE TABLE summaries (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    uuid REFERENCES sessions(id),
  parent_id     uuid REFERENCES summaries(id),  -- encadeamento
  content       text NOT NULL,
  token_count   int NOT NULL,
  covers_until  int NOT NULL,              -- maior turn.idx coberto
  created_at    timestamptz DEFAULT now()
);

-- checkpoint: classificação de assunto a cada 20 turns de user
CREATE TABLE checkpoints (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    uuid REFERENCES sessions(id),
  user_id       uuid REFERENCES users(id),
  at_user_turn  int NOT NULL,              -- 20, 40, 60...
  label         text NOT NULL,              -- "poucas palavras" do assunto
  created_at    timestamptz DEFAULT now()
);

-- skills por usuário (autoradas ou induzidas)
CREATE TABLE skills (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id       uuid REFERENCES users(id),
  name          text NOT NULL,
  summary       text NOT NULL,             -- 1 linha p/ GetSkills / progressive disclosure
  body          text NOT NULL,             -- procedimento completo (carregado sob demanda)
  origin        text DEFAULT 'induced',     -- authored | induced
  embedding     vector(1536),               -- busca semântica (pgvector)
  created_at    timestamptz DEFAULT now()
);

-- índice de Index Tools (vindas de MCP) — NÃO entram no system prompt
CREATE TABLE tool_index (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  mcp_server    text NOT NULL,
  name          text NOT NULL,
  description   text NOT NULL,
  input_schema  jsonb NOT NULL,
  embedding     vector(1536),               -- SearchTools busca aqui
  enabled       bool DEFAULT true
);

-- log de observabilidade: um registro por step do loop
CREATE TABLE step_logs (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    uuid REFERENCES sessions(id),
  turn_id       uuid REFERENCES turns(id),
  step_type     text NOT NULL,              -- model_call | tool_call | summarize | checkpoint
  detail        jsonb,
  tokens_in     int, tokens_out int,
  latency_ms    int,
  created_at    timestamptz DEFAULT now()
);

04Provider · OpenRouter

OpenRouter dá três coisas que o harness precisa: o tamanho da janela de contexto por modelo, o endpoint de chat completions com tool calling, e o usage com tokens de entrada/saída por chamada.

1 · Metadata do modelo (janela de contexto)

No início da session, buscamos GET /api/v1/models e lemos o context_length do modelo escolhido. Esse valor alimenta sessions.context_window e é a base do orçamento de memory.

2 · Chat completions + tool calling

class OpenRouterProvider:
    def model_context_window(self, model: str) -> int:
        # cacheado por session; fonte: /api/v1/models -> context_length
        return self._models[model]["context_length"]

    def complete(self, model, messages, tools) -> ModelResult:
        r = httpx.post("https://openrouter.ai/api/v1/chat/completions",
            headers={"Authorization": f"Bearer {API_KEY}"},
            json={"model": model, "messages": messages,
                  "tools": tools, "tool_choice": "auto"})
        d = r.json()
        return ModelResult(
            message    = d["choices"][0]["message"],
            tokens_in  = d["usage"]["prompt_tokens"],      # grava no turn
            tokens_out = d["usage"]["completion_tokens"])
Contabilidade de tokens — fonte da verdade Use sempre o usage retornado pelo OpenRouter para gravar tokens_in/tokens_out no turn — é o número real cobrado. Para estimar antes da chamada (ex.: caber na janela), use um tokenizer local (tiktoken ou o do modelo) só como aproximação de token_count.

05Loop de execução

Ciclo agêntico com guard de orçamento de tokens por session. Ao estourar o teto, para e devolve o que tiver.

1
Recebe a mensagem do usuário → grava turn role=user (incrementa contador de user-turns para checkpoint).
2
Monta o contexto via Memory: system prompt + summary + janela de turns ativos, sempre dentro do orçamento.
3
Chama o modelo (OpenRouter). Grava turn role=assistant com tokens_in/out; soma em sessions.tokens_spent.
4
Tem tool call? Executa a tool (built-in ou Index Tool), grava turn role=tool e volta ao passo 2.
5
Sem tool call → resposta final ao usuário. Encerra o turn.
6
Guards a cada iteração: orçamento de tokens, teto de iterações, gatilho de summary e gatilho de checkpoint.
def run_turn(session, user_msg):
    memory.append(session, role="user", content=user_msg)
    memory.maybe_checkpoint(session)        # a cada 20 user-turns

    for step in range(MAX_STEPS):
        # --- GUARD de orçamento: para e retorna parcial ---
        if session.tokens_spent >= session.token_budget:
            session.status = "budget_exhausted"
            return PartialResult(memory.last_assistant_text(session),
                                 reason="token_budget")

        ctx = memory.build_window(session)     # system + summary + turns ativos
        res = provider.complete(session.model, ctx, tools.builtins())

        turn = memory.append(session, role="assistant",
                            content=res.message,
                            tokens_in=res.tokens_in, tokens_out=res.tokens_out)
        session.tokens_spent += res.tokens_in + res.tokens_out
        observer.log(session, turn, "model_call", res)

        memory.maybe_summarize(session)       # se janela > orçamento de memory

        if not res.message.tool_calls:
            return FinalResult(res.message.content)

        for call in res.message.tool_calls:
            out = tools.dispatch(session, call)    # builtin OU index tool
            memory.append(session, role="tool", content=out)
            observer.log(session, turn, "tool_call", out)
Reserva de saída Sua regra é memory = janela − system prompt. Na prática, reserve também espaço para a resposta do modelo: orçamento_memory = context_window − system_prompt − reserva_resposta. Sem isso, um contexto cheio não deixa tokens para o modelo responder. Sugiro reserva_resposta configurável (ex.: 2.000–4.000 tokens).

06Memory

A janela é dirigida por orçamento de tokens. Quando enche, sumariza encadeado e mantém só as 10% mensagens mais recentes verbatim. O histórico completo nunca se perde — fica no Postgres.

O cálculo do orçamento

# no início da session
context_window   = provider.model_context_window(model)   # ex. 200_000
system_tokens    = count_tokens(system_prompt)
memory_budget    = context_window - system_tokens - response_reserve

# janela ativa atual (o que será enviado ao modelo)
window_tokens    = summary.token_count + sum(t.token_count
                       for t in active_turns)   # turns com in_window=true

Gatilho de sumarização

Quando window_tokens ≥ memory_budget, dispara o procedimento:

1
Copia as 10% últimas mensagens (keep = ceil(0.10 × nº de turns ativos)). Elas permanecem na janela, verbatim.
2
Gera um summary encadeado de todo o memory (exceto system prompt): inclui o summary anterior + todos os turns que sairão da janela.
3
Marca os turns sumarizados com in_window=false (continuam no DB, fora da janela ativa).
4
Nova janela = system prompt + summary + 10% recentes.
def maybe_summarize(session):
    active = turns_in_window(session)
    window = current_summary_tokens(session) + sum(t.token_count for t in active)
    if window < session.memory_budget:
        return

    keep_n   = ceil(0.10 * len(active))
    keep     = active[-keep_n:]                 # 10% recentes ficam verbatim
    to_fold  = active[:-keep_n]                 # o resto é sumarizado

    prev     = current_summary(session)         # encadeamento
    new_text = provider.summarize(prev_summary=prev, messages=to_fold)

    summaries.insert(session, parent=prev, content=new_text,
                     covers_until=to_fold[-1].idx)
    mark_out_of_window(to_fold)                 # in_window=false (continuam no DB)
    observer.log(session, None, "summarize", {"folded": len(to_fold)})

Checkpoints de assunto

Contamos turns de role user. A cada 20, classificamos em poucas palavras o assunto tratado desde o checkpoint anterior — uma trilha navegável da session, barata em tokens.

def maybe_checkpoint(session):
    n = count_user_turns(session)
    if n > 0 and n % 20 == 0:
        recent = user_turns_since_last_checkpoint(session)
        label  = provider.classify_subject(recent)   # "poucas palavras"
        checkpoints.insert(session, at_user_turn=n, label=label)
        observer.log(session, None, "checkpoint", {"label": label})
Por que isto funciona O DB é a fonte autoritativa do histórico (todos os turns, com in_window). A janela ativa é só uma projeção dentro do orçamento. Sumarizar não apaga nada: transforma "lembrar tudo" em "reconhecer um resumo + o presente recente" — exatamente a transformação de recall → retrieval do artigo de referência.

07Tools & MCP

Duas estratégias juntas. As built-in ficam sempre no system prompt (são poucas). As Index Tools (de MCP) podem ser milhares — ficam num índice buscável e só entram no contexto quando o agente as procura.

🧱Built-in tools (sempre no prompt)

GetTools retorna o schema completo de uma tool específica do índice.
SearchTools busca semântica no índice de Index Tools por descrição da necessidade.
GetSkills lista/recupera skills do usuário (progressive disclosure).
Bash executa comandos atrás do SandboxBackend plugável.

🔌Index Tools (via MCP)

Cada servidor MCP conectado tem suas tools registradas em tool_index (nome, descrição, schema, embedding). Nenhuma entra no system prompt por padrão — isso preservaria zero espaço para a memory. O agente as descobre por SearchTools e carrega o schema por GetTools só quando precisa.

Fluxo de descoberta → uso

 modelo precisa de uma capacidade que não tem no prompt
        │
        ▼
  SearchTools("enviar email para cliente")
        │   busca vetorial em tool_index
        ▼
  retorna top-k: [{name, description}, ...]   ← barato (só resumos)
        │
        ▼
  GetTools("gmail.send")  → schema completo da tool
        │
        ▼
  modelo emite tool_call ─▶ dispatch ─▶ MCP server ─▶ resultado
        │
        ▼
  resultado normalizado vira turn role=tool na memory
class ToolRegistry:
    BUILTINS = ["GetTools", "SearchTools", "GetSkills", "Bash"]

    def builtins(self) -> list:  # só estas vão ao system prompt
        return [self._spec(n) for n in self.BUILTINS]

    def search_tools(self, query, k=8):
        emb = embed(query)
        return db.query("""SELECT name, description FROM tool_index
                            WHERE enabled ORDER BY embedding <=> %s LIMIT %s""",
                         [emb, k])

    def dispatch(self, session, call):
        if call.name in self.BUILTINS:
            return self._run_builtin(session, call)
        return mcp.invoke(call.name, call.arguments)   # Index Tool via MCP
A economia de contexto Com N tools de MCP, o custo no prompt é O(1) (só as 4 built-ins), não O(N). O agente "paga" tokens de descrição de tool apenas pelas que efetivamente busca — preservando o orçamento para a memory.

08Skills

Skills pertencem ao usuário e vivem no Postgres. Além das autoradas, o harness induz skills novas observando padrões nos pedidos — a cada 10 sessions.

Acesso em runtime

O agente descobre skills via a built-in GetSkills (progressive disclosure: primeiro os summary de uma linha; o body completo só é carregado quando a skill é escolhida). Busca semântica por embedding quando o usuário tem muitas skills.

Indução automática (a cada 10 sessions)

Quando o usuário fecha sua 10ª, 20ª, 30ª… session, dispara um job (assíncrono) que analisa o histórico de pedidos e procura padrões recorrentes que valham virar skill.

1
Gatilho: ao fechar uma session, count_closed_sessions(user) % 10 == 0.
2
Coleta: reúne sinais baratos — turns role=user, labels de checkpoints e summaries das últimas sessions.
3
Detecção: uma chamada ao modelo identifica padrões repetidos que poderiam virar procedimento reutilizável.
4
Persistência: se encontrar, insere uma nova skill com origin='induced' para o usuário (dedup por similaridade de embedding com as existentes).
def on_session_closed(session):
    user = session.user_id
    if count_closed_sessions(user) % 10 != 0:
        return

    signals = gather_recent_requests(user)        # user-turns + checkpoints + summaries
    found   = provider.induce_skills(signals)     # -> [] ou [SkillDraft, ...]

    for draft in found:
        if not is_duplicate(user, draft.embedding):  # evita repetir skill existente
            skills.insert(user_id=user, name=draft.name,
                          summary=draft.summary, body=draft.body,
                          origin="induced", embedding=draft.embedding)
            observer.log(session, None, "skill_induced", {"name": draft.name})
Custo controlado A indução roda 1× a cada 10 sessions, fora do caminho crítico do loop (assíncrona). Alimentá-la com checkpoints e summaries (já comprimidos) em vez do histórico bruto mantém a chamada barata em tokens.

09Observabilidade

Todo step do loop é logado; cada turn carrega o consumo real de tokens. Token-spend é cidadão de primeira classe.

📝Log por step

Um registro em step_logs para cada model_call, tool_call, summarize e checkpoint — com detail, latency_ms e tokens.

🔢Tokens no turn

tokens_in/tokens_out gravados em cada turn gerado pelo modelo (fonte: usage do OpenRouter); token_count em todos.

📈Agregados por session

sessions.tokens_spent mantém o acumulado vivo — é o que o guard do loop consulta para parar no teto.

🔍Trilha auditável

Com step_logs + turns + checkpoints, qualquer session é reconstruível passo a passo para debug e custo.

Métricas derivadas direto do SQL: tokens por turn, por session e por usuário; custo médio por resposta; número de sumarizações; latência por tipo de step.

-- gasto de tokens por usuário no mês
SELECT u.external_id,
       SUM(t.tokens_in)  AS input,
       SUM(t.tokens_out) AS output
FROM turns t JOIN users u ON u.id = t.user_id
WHERE t.created_at >= date_trunc('month', now())
GROUP BY u.external_id ORDER BY output DESC;

10Roadmap em fases

Cada fase entrega algo executável. As fases 1–3 já dão um harness funcional; 4–6 adicionam a inteligência de memory/skills.

Fase 1 · Fundação
  • Esquema Postgres (tabelas acima) + migrações; habilitar pgvector.
  • OpenRouterProvider: metadata de modelo, complete(), captura de usage.
  • Modelo de session/turn com gravação de tokens.
Fase 2 · Loop mínimo
  • Loop perceber→chamar→parsear→responder com built-in Bash atrás de SandboxBackend (impl. trivial p/ começar).
  • Guard de orçamento por session: parar e retornar parcial.
  • Observer gravando step_logs.
Fase 3 · Tools & MCP
  • Cliente MCP + ingestão de tools no tool_index (com embeddings).
  • Built-ins SearchTools e GetTools; dispatch para Index Tools.
Fase 4 · Memory completa
  • Orçamento de memory, build_window, maybe_summarize (encadeado + 10% verbatim).
  • maybe_checkpoint a cada 20 user-turns.
Fase 5 · Skills
  • GetSkills com progressive disclosure; skills autoradas.
  • Job de indução em on_session_closed (a cada 10 sessions) com dedup.
Fase 6 · Endurecimento
  • Trocar o SandboxBackend trivial por isolamento de kernel (gVisor/Firecracker) — ver plano de sandbox.
  • Dashboards de tokens/custo; quotas; alertas de gasto.
