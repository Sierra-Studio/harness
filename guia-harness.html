● Harness Engineering for LLM Agents

How to build your own harness — pluggable, isolated and scalable

An implementation guide derived from “Externalization in LLM Agents: A Unified Review of Memory, Skills, Protocols and Harness Engineering” (Zhou et al., 2026), combined with the architecture patterns from the Hermes Agent. It covers every element of a powerful harness, multi-tenancy with isolated sandboxes, secure transport, and deployment on Kubernetes or a VM.

00What a harness is (and isn't)

Agent capability does not live only in the model — it emerges from the coupling between the model and the environment that organizes its cognition into action.

The paper makes a central claim: modern agents are built less by changing model weights and more by reorganizing the runtime around them. Capabilities we once expected the model to recover internally are now externalized into four layers:

🧠Memory

Externalizes state across time — turns "recall" into "retrieval".

🛠️Skills

Externalizes procedural expertise — turns improvised generation into guided composition.

🔌Protocols

Externalizes interaction structure — turns ad-hoc coordination into structured exchange.

⚙️Harness

The unification layer that coordinates everything above into governed execution.

Operational definition A harness is the designed cognitive environment within which the externalized modules become jointly effective. It governs what the model perceives, what it retains across turns, which operations are callable, which actions require approval, which intermediate states are exposed for revision, and which failures are detectable and recoverable. It defines the agent's practical cognitive boundary.

In practice a harness is not "a synonym for a memory system" nor "tool calling rebranded". It is the discipline of building the representational, procedural and operational conditions under which the model's reasoning becomes stable enough to sustain real work.

02The 3 externalization modules

These are the agent's cognitive content. Distinct yet coupled — each one feeds the others in a self-reinforcing cycle.

1
Module · Externalized state

🧠 Memory — state persistence

Carries continuity across time: session state, project context and failure recording. It turns the recall problem into a targeted retrieval problem — instead of flooding the model with the full log, it selects the slice relevant to the current step.

  • What it externalizes: facts, episodes (past trajectories), entity relations, cross-session context.
  • Architectures: monolithic context → context + retrieval (RAG) → hierarchical/orchestrated memory → adaptive memory.
  • Governed paths: every read/write must be inspectable, auditable and versioned (provenance).
  • Key risk: memory poisoning — corrupted traces silently distort future reasoning.
SummarizationEpisodic tracesVector storeProvenanceCross-session recall
2
Module · Externalized expertise

🛠️ Skills — reusable routines

Package procedural expertise: operational procedures, decision heuristics and normative constraints. The model is asked less often to invent a workflow from scratch and more often to interpret and follow a prepared one.

  • Lifecycle: specification → discovery → progressive disclosure (staged loading) → execution binding → composition.
  • Acquisition: distilled from successful trajectories (memory→skill) and revised from recurring failures (skill→memory).
  • Hermes pattern: autonomous skill creation after complex tasks; skills that self-improve during use; compatible with the open agentskills.io standard.
  • Key risk: malicious skill injection and tool sprawl that turns action selection into a disambiguation problem.
Staged loadingSKILL.mdFailure-driven revisionCompositionSkills Hub
3
Module · Externalized interaction

🔌 Protocols — deterministic interfaces

They replace open-ended natural-language inference with a structured task: fill typed fields, follow a declared state transition and receive structured feedback. The model contributes judgment; the protocol contributes format, validation and lifecycle control.

  • Types: Agent↔Tool (MCP, OpenAPI, JSON-RPC), Agent↔Agent (A2A), Agent↔User.
  • Role in the harness: intent capture/normalization, capability discovery, tool description, session and lifecycle management.
  • As action schema: shape the output boundary — constrain the generative space (JSON Schema) and make execution deterministic enough to govern.
  • Key risk: protocol spoofing — forged manifests or manipulated endpoints cause unauthorized actions under a legitimate appearance.
MCPJSON SchemaOpenAPIA2ASchema validation
The self-reinforcing cycle Memory feeds skill distillation; skills produce rich traces; traces improve memory; protocols normalize results back into memory (Protocol→Memory). Beware: this positive feedback also amplifies errors — a poisoned memory entry yields a flawed skill, whose traces further contaminate memory. No single module can interrupt the cascade without harness-level intervention.

03The 6 operational dimensions

These are not forms of externalization — they are the coordinating infrastructure that makes memory, skills and protocols work as a coherent system. Independently developed systems (Codex, Claude Code) converge on these six, which signals they are structural requirements, not incidental choices.

4
Control · Temporal backbone

Agent Loop & Control Flow

The perceive → retrieve → plan → act → observe → revise loop. What the harness adds beyond the bare loop is governance over termination, recursion and resource consumption — without it, the loop cycles indefinitely, escalates costs through unbounded tool calls, or recurses into sub-agents that exhaust the budget.

  • Topologies: single-loop (reasoning+action in one pass), hierarchical (planner + executors), multi-agent (subtasks routed to agents with distinct tool sets and permission scopes).
  • Mandatory controls: max step count, recursion depth limit, per-step cost ceiling, timeouts. They define the operational envelope.
5
Permission · Execution boundary

Sandboxing & Execution Isolation critical for this project

Whenever the agent acts on the world — writing files, executing shell, calling APIs — the harness decides how much of the environment to expose and how to contain side effects. The sandbox is not just a security fence: it is a cognitive boundary that simplifies the environment by removing irrelevant state and making the workspace inspectable.

  • Granularities: a dedicated sandbox per task (its own filesystem snapshot, network restrictions, resource quotas) — the Codex model; or graduated permission modes (autonomous → mandatory approval per tool call) — the Claude Code model.
  • Principle: one execution can never contaminate another. This is the foundation of the multi-tenant isolation in §05.
+
Permission · Human oversight

Human Oversight & Approval Gates

Full autonomy is rarely appropriate. The level of autonomy is a configurable harness parameter — adjustable per task, per tool and per organizational policy — not a binary property of the agent.

  • Pre-execution approval: pauses before each consequential action and asks for confirmation.
  • Post-execution review: lets the agent act but surfaces results before committing/continuing.
  • Escalation triggers: runs autonomously but halts on risk signals (sensitive data, irreversible operation, confidence below a threshold).
  • Hook systems: attach arbitrary logic (scripts, validations, notifications) to lifecycle events — tool invocation, file write, subagent spawn.
6
Observability · Structured feedback

Observability & Structured Feedback

An agent that acts without leaving inspectable traces cannot be debugged, audited or improved. Observability serves two purposes: externally (debugging, compliance auditing, post-incident analysis) and internally (it closes the feedback loop that connects execution outcomes back to the modules).

  • What to log: every model invocation, tool call, memory read/write and decision branch; execution traces linking each action to its causal antecedents.
  • Aggregate metrics: step counts, token consumption, error rates, latency distributions.
  • Internal feedback: failed tool call → memory write; repeated failures → flag a skill for revision; latency spike → switch protocol path.
5
Permission · Externalized governance

Configuration, Permissions & Policy Encoding

The harness encodes not only what the agent can do, but what it is allowed to do and under what conditions. Configuration separates policy from execution logic and makes the rules explicit, versionable and auditable.

  • Stratified scopes: user preferences/trust → project available tools, accessible paths, commands requiring approval → org compliance, cost ceilings, data rules projects cannot override.
  • Governance as data: declarative rules enforced at runtime, not embedded in prompts nor filtered post-hoc. The same base agent runs under different policy regimes without changing the model or its skills.
7
Control · The scarcest resource

Context Budget Management

The context window is the scarcest shared resource. Memory retrieval, skill loading, protocol schemas, tool descriptions and the reasoning traces themselves all compete for the same finite token budget. It is a dynamic resource-allocation problem, not isolated compression.

  • Summarization: compresses older history while preserving decision-relevant information.
  • Priority-based eviction: removes/demotes entries whose relevance to the active subtask has decayed.
  • Staged loading: detailed procedural guidance enters the context only when a matching task pattern is detected.
  • Budget-aware routing: the optimal allocation depends on the phase — early planning wants more memory; late execution wants more skill.
Design principle The goal is efficient, utility-positive externalization, not maximal. Minimal sufficiency: each module must reduce the cognitive burden, not add one. A good harness simplifies the model's decision problem; it does not create a second one.

04Pluggable architecture (Hermes-style)

To swap the harness or the sandbox without rewriting the system, everything communicates through stable interface contracts. Hermes does exactly this: it is model-agnostic, has a modular toolset system, and supports multiple terminal backends (local, Docker, SSH, Singularity, Modal, Daytona) behind the same interface.

The key to pluggability is defining four contract boundaries and programming against them, never against concrete implementations:

①Harness ↔ Model

ModelProvider interface. Any LLM (Claude, GPT, open-weights) pluggable via an adapter that normalizes calls and tool-calling.

②Harness ↔ Sandbox

SandboxBackend interface. Docker, gVisor, Firecracker, K8s Pod, Daytona, Modal — all behind the same exec/read/write contract.

③Harness ↔ Tools

Toolset interface + MCP. Native tools and any external MCP server discovered via capability discovery.

④Harness ↔ Frontend

Transport gateway. A single agent logic serving CLI, Web, Slack, etc. (Hermes' "gateway process").

Contract 1 — HarnessEngine (swappable)

The whole harness sits behind a facade. You can replace the entire engine (e.g. swap your implementation for a Hermes-based one) while keeping sandbox, tools and frontend.

// The contract ANY harness must fulfill. Swapping the harness = swapping the impl.
interface HarnessEngine {
  // starts/resumes a per-user isolated session (see §05)
  createSession(ctx: SessionContext): Promise<SessionHandle>

  // runs one turn of the agent loop; emits observable events
  step(session: SessionHandle, input: UserInput): AsyncIterable<AgentEvent>

  // pluggable dependency injection
  model:    ModelProvider
  sandbox:  SandboxBackend
  tools:    ToolRegistry
  memory:   MemoryStore
  policy:   PolicyEngine
  tracer:   Observer
}

Contract 2 — SandboxBackend (swappable)

The harness never runs bash directly on the host. It delegates to a backend that fulfills this contract. Swapping Docker for Firecracker or for a K8s Pod is swapping the implementation, not the harness.

interface SandboxBackend {
  // provisions an isolated environment for ONE session of ONE user
  provision(spec: SandboxSpec): Promise<Sandbox>

  // runs a command inside the isolation; streams stdout/stderr
  exec(sb: Sandbox, cmd: Command): AsyncIterable<ExecChunk>

  readFile(sb: Sandbox, path: string): Promise<Bytes>
  writeFile(sb: Sandbox, path: string, data: Bytes): Promise<void>

  // destroys ALL state — guarantees nothing leaks to the next session
  destroy(sb: Sandbox): Promise<void>
}

type SandboxSpec = {
  sessionId:  string
  userId:     string          // used for namespacing and quotas
  image:      string          // minimal base image
  cpu:        string          // e.g. "500m"
  memory:     string          // e.g. "512Mi"
  timeoutSec: number
  network:    "none" | "egress-allowlist"
  ttlSec:     number          // auto-destruction
}
The golden rule of pluggability If the harness knows the name Docker or Firecracker anywhere outside the backend factory, the abstraction has leaked. The harness only knows SandboxBackend. The concrete choice comes from configuration: backend: kubernetes-pod | firecracker-vm | docker | daytona.

05Sandbox & multi-tenant isolation

The core requirement of this project: many users on the same harness, with no user able to access another's session. Every session that needs bash runs in a disposable sandbox with resource isolation.

The model: one ephemeral sandbox per session

The harness is a shared service that is stateless with respect to execution; the dangerous state (filesystem, processes, network) always lives inside a dedicated sandbox, created for one session and destroyed at the end. This gives the three "I"s: Isolation (one execution does not contaminate another), Identity (each sandbox is bound to a userId) and Impermanence (TTL + destroy guarantee nothing leaks).

Important — kernel isolation, not just containers For true multi-tenancy with arbitrary code execution, Docker namespaces are not enough. Use a kernel boundary: gVisor (runsc), Kata Containers or Firecracker microVMs. They guarantee that an escape in A's sandbox cannot reach the host nor B's sandbox. Daytona and Modal (Hermes backends) already deliver this isolation as a service.

Session lifecycle

// pseudo-flow inside the HarnessEngine
async function handleTurn(userId, sessionId, input) {
  const sb = await registry.sandbox.provision({
    sessionId, userId,
    image: "harness/base-tools:slim",
    cpu: "500m", memory: "512Mi",
    network: "egress-allowlist",
    timeoutSec: 120, ttlSec: 1800
  })
  try {
    for await (const ev of engine.step(session, input)) {
      if (ev.type === "tool:bash")
        await registry.sandbox.exec(sb, ev.cmd)   // runs INSIDE the isolation
      tracer.emit(ev)                              // observability (§03.6)
    }
  } finally {
    await registry.sandbox.destroy(sb)            // nothing survives the turn
  }
}

06Secure data path: harness ↔ sandbox

The channel between control plane and sandbox carries commands, file contents and execution outputs — potentially sensitive data. It must be encrypted, mutually authenticated and minimally scoped.

🔒mTLS

Mutual TLS between harness and each sandbox. Both sides present a certificate; no cleartext traffic, even inside the cluster.

🎫Scoped tokens

Each sandbox gets a short-lived token valid only for {userId, sessionId}. No shared ambient credentials.

📡Single channel

All communication over one structured protocol (gRPC/WebSocket over mTLS). No ad-hoc SSH nor extra open ports.

Recommended pattern

  • Transport: gRPC with bidirectional streaming over mTLS — commands go down, ExecChunk (stdout/stderr/exit) streams up. On K8s, optionally a service mesh (Istio/Linkerd) provides automatic mTLS and SPIFFE/SPIRE identity per workload.
  • Authentication: workload identity (SPIFFE ID) per sandbox; the harness validates that the ID matches the expected sessionId before accepting any byte.
  • Authorization: the session token limits which operations the channel accepts (command allowlist, permitted paths). Least privilege on the wire itself.
  • Data confidentiality: sensitive data in transit encrypted by mTLS; at rest (if scratch is persisted) encrypted at the volume level. Logs/traces (§03.6) redact secrets before persisting.
  • Integrity: protocol results are normalized and assimilated back into memory (Protocol→Memory) only after schema validation — preventing protocol spoofing.

07Deployment: Kubernetes or VM

The same architecture runs on both targets because the sandbox backend is pluggable. On K8s, each sandbox is an ephemeral Pod; on a single VM, each sandbox is a microVM or a container with kernel isolation.

  • Sandbox = ephemeral Pod with a kernel-isolation RuntimeClass (runsc for gVisor or kata), created by the Orchestrator and removed at session end.
  • Isolation: NetworkPolicy default-deny + dedicated namespace + ResourceQuota + seccomp/AppArmor + non-root + read-only root fs.
  • Scale: HPA scales the control plane; sandboxes scale naturally (1 per session). A warm pool can reduce cold-start latency.
  • mTLS: service mesh (Istio/Linkerd) or cert-manager + SPIRE for per-workload identity.

Option B — Single VM (smaller footprint)

  • Sandbox = microVM (Firecracker) or container with gVisor, orchestrated by a local daemon that implements the same SandboxBackend.
  • The harness runs as a service on the VM; communication over a local socket on mTLS. Kernel isolation remains the security boundary.
  • Hermes shows this runs "on a $5 VPS or on a GPU cluster" — the interface does not change; only the configured backend does.

08Threats & governance

Externalization is not free: by moving cognitive burden into external artifacts, those artifacts become targets. Governance is a co-designed layer of the harness, not an afterthought.

☠️Lethal Trifecta

Sensitive-data access + unconstrained external communication + unverified execution. Break at least one of the three by design.

🧪Memory poisoning

Corrupted traces distort future reasoning. Mitigate with provenance, write validation and deterministic rollback.

💉Skill injection

Adversarial procedures in the repertoire. Mitigate with mandatory review gates for critical skills and artifact signing.

🎭Protocol spoofing

Forged manifests / manipulated endpoints. Mitigate with skill-independent schema validation and mTLS.

🏚️Tool sprawl

Too many tools becomes a disambiguation problem. Mitigate with scoped discovery and staged loading.

🌊Error cascade

The self-reinforcing cycle amplifies failures. Only harness-level intervention interrupts it — hence observability.

Governance as infrastructure Mandatory review gates for critical updates · provenance tracking for memory and skill changes · deterministic rollback mechanisms · regression testing. The quality of an externalized system is measured not only by what it enables, but by how transparently and reversibly it does so.

09Implementation checklist

A minimal roadmap for a pluggable, multi-tenant and secure harness. Each item maps to a section above.

Externalization modules

  • Memory with governed read/write paths, provenance and selective retrieval §02
  • Skills with staged loading (progressive disclosure) and failure-driven revision §02
  • Protocols typed (MCP/JSON Schema) with validation at the output boundary §02

Operational dimensions

  • Agent loop with step limits, recursion depth, cost ceiling and timeout §03
  • Sandboxing with kernel isolation (gVisor/Kata/Firecracker) per session §03·05
  • Approval gates configurable (pre/post/escalation) + hook system §03
  • Observability: structured logging, execution traces and aggregate metrics §03
  • Policy engine stratified (user/project/org), declarative and versioned §03
  • Context budget: summarization + priority eviction + staged loading §03

Pluggability & infrastructure

  • 4 interface contracts: ModelProvider, SandboxBackend, Toolset/MCP, Frontend gateway §04
  • Ephemeral sandbox per session bound to userId, with TTL + idempotent destroy §05
  • 6 isolation layers: identity, compute, fs, network, resources, time §05
  • mTLS transport + scoped tokens + workload identity (SPIFFE) §06
  • Selectable backend by config: kubernetes-pod | firecracker-vm | docker-gvisor | daytona §07

Governance

  • Review gates + provenance + deterministic rollback + regression testing §08
  • Defense against the Lethal Trifecta: break data+communication+unconstrained execution §08
  • Secret redaction in logs and traces before persisting §06·08
Suggested next step Start with the 4 interface contracts (§04) using trivial implementations (sandbox = local Docker, model = one provider, frontend = CLI). With the contracts in place, you can swap each piece — including the entire harness — without rewriting the rest. Then harden isolation (gVisor → Firecracker) and transport (mTLS) before opening up to multiple users.
